# ADR-001: Manual MCP First, Hooks Opt-In

Generated by /office-hours on 2026-06-08
Repo: claude-for-codex
Status: APPROVED
Mode: Builder

## Problem Statement

The team wants Codex users to be able to call Claude Code for review and rescue
work without moving their primary workspace out of Codex. The product should
serve as the Codex-first counterpart to `openai/codex-plugin-cc`: Claude Code is
the outside reviewer, not the main working surface.

The near-term audience is not all Codex users. The first audience is teammates
who use Codex, including some who also use Claude Code and some who do not. The
observed team behavior is that AI-generated work often receives no AI second
opinion at all.

## What Makes This Useful

The useful wedge is not a fully automated review system or another planning
framework. The useful wedge is a low-friction manual escape hatch inside Codex:

- Ask Claude to review a Codex diff.
- Ask Claude to pressure-test a risky change.
- Ask Claude to investigate or rescue a stuck task.

This gives Codex-first users a second model when they want one, without forcing
every turn through an automated gate or requiring a Claude-first workflow.

## Constraints

- The first version should serve team-local adoption before public plugin scale.
- The default path must be explicit and predictable for teammates.
- The product boundary is a Codex-to-Claude model bridge, not a GSD/gstack
  replacement, PR review bot, or full team process framework.
- Read-only review should be the default safety posture.
- Any write-capable Claude delegation must be opt-in.
- Lifecycle hooks are powerful but surprising; they must not be enabled by
  default.
- Hook setup must have clear enable, disable, and status/verification commands.
- Current Codex docs use `features.hooks` as the canonical feature key;
  `features.codex_hooks` is deprecated.
- The MCP server runtime is Node.js. Go and Rust are deferred until single-binary
  distribution becomes more important than fast MCP iteration.

## Premises

1. The team currently lacks a consistent review habit for Codex work.
2. A manual command is more acceptable than an automatic stop-time gate for the
   first rollout.
3. Hook-based automation should be treated as an advanced mode because it changes
   Codex lifecycle behavior.
4. Team adoption depends more on setup clarity than on adding many Claude modes.
5. The plugin should make review easy first, then make automation available when
   the team explicitly wants enforcement.

## Approaches Considered

### Approach A: Manual Review Only

Expose `/claude-review` and `/claude-adversarial` as the primary team workflow.
This is the smallest and safest product surface.

Pros:
- Lowest setup and trust burden.
- Easy for teammates to understand.
- Keeps Claude read-only by default.

Cons:
- Does not help if teammates forget to ask for review.
- Less useful when someone wants Claude to continue investigation work.

### Approach B: Manual Review Plus Rescue

Expose review tools plus `/claude-rescue`, with rescue read-only by default and
write access behind an explicit option.

Pros:
- Review becomes part of a broader "second model" workflow.
- More likely to become a repeated habit because it helps during stuck work.
- Still avoids surprising lifecycle automation.

Cons:
- Requires clearer safety language around write permissions.
- Background job status/result UX matters more.

### Approach C: Hook-First Review Gate

Install a `Stop` hook that asks Claude to review before Codex finishes a turn.
Blocking findings can prevent Codex from stopping.

Pros:
- Creates review by default.
- Useful once a project wants enforcement.
- Can catch issues even when the user forgets to ask.

Cons:
- Surprising for first-time users.
- Can slow turns and drain usage limits.
- Multiple hook sources can all run; debugging behavior is harder.

### Approach D: Packaging First

Prioritize installer, config validation, status checks, and team README before
adding more model workflows.

Pros:
- Reduces team setup failures.
- Makes adoption measurable and repeatable.
- Gives clear enable/disable/status paths.

Cons:
- Less exciting than model features.
- Still needs a useful core workflow to justify installation.

## Decision

Use Approach B with the packaging discipline from Approach D.

Manual MCP tools and slash prompts are the default product:

- `claude_review`
- `claude_adversarial_review`
- `claude_rescue`
- `claude_status`
- `claude_result`
- `claude_cancel`
- `claude_setup`

Hooks are supported, documented, and testable, but disabled by default. The hook
experience should be explicitly opt-in:

- `setup --enable-hook`
- `setup --disable-hook`
- `setup --status`

The runtime decision is Node MCP:

- Keep the MCP server in Node.js.
- Move the server body from prototype `.mjs` to TypeScript before team rollout.
- Keep small hook/setup scripts in `.mjs` when that remains the simplest form.
- Revisit Go only if installer/status distribution becomes the bottleneck.
- Revisit Rust only if a security-sensitive single-binary product becomes the
  target.

## Recommended Product Shape

Default install:

- Registers the MCP server.
- Installs optional Codex prompt files.
- Verifies `claude` is installed and authenticated enough to answer
  `claude --version`.
- Does not configure any lifecycle hook.

Optional hook install:

- Adds a `Stop` hook that runs `hooks/review-gate.mjs`.
- Uses `[features] hooks = true`, not the deprecated `codex_hooks` alias.
- Explains that hooks can also be inspected and disabled through `/hooks`.
- Provides a disable path that removes or comments out only this plugin's hook
  block.

## Success Criteria

- A teammate can install the plugin and run `claude_setup`.
- A teammate can invoke `/claude-review` or ask Codex to use `claude_review`.
- The default workflow never blocks Codex from stopping.
- A teammate can opt into the hook and verify it is active.
- A teammate can disable the hook without uninstalling the MCP server.
- Docs clearly distinguish "plugin installed" from "hook enabled".

## Distribution Plan

Team-local distribution should start as a Git repository with:

- `npm install` for dependencies.
- A setup script that edits or prints Codex config snippets.
- A status command that validates Node, Claude Code, config shape, and prompt
  installation.
- Documentation that explains manual mode first and hook mode second.

Public packaging can come later once the team-local install path is stable.

## Next Steps

1. Update `README.md` and `docs/SETUP.md` so manual MCP setup is the default path.
2. Replace `codex_hooks = true` examples with `hooks = true`.
3. Add setup/status language for hook enable, hook disable, and hook inspection.
4. Convert the MCP server body to Node + TypeScript before team rollout.
5. Add a small installer or helper script only after the manual docs are correct.
6. Verify with a fresh local Codex config parse and a `claude_setup` tool call.

## What I Noticed

- "리뷰 자체를 안 하고 있는 것 같아" means the product should create a review
  habit, not optimize an existing one.
- "자동 리뷰를 하게끔 만들고 싶지는 않고" sets the default: no automatic gate.
- "hook에 대한 이해가 아직 덜 되어서" is a documentation requirement, not just
  a product question. Hook behavior must be explained in plain operational terms.
