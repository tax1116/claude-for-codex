# Plan 01-02: Slash-Command Team Rollout

<objective>
As a teammate adopting `claude-for-codex`, I want setup and docs to teach the
manual slash-command path first, while still exposing MCP tool names as the
reference interface, so that I can choose the right Claude review mode without
turning on automation or misunderstanding what context Claude sees.
</objective>

<must_haves>
- D-01: Docs and prompts must expose `/claude-review` and `/claude-adversarial` as the two primary team commands.
- D-02: Docs and prompts must show that no arguments are required and `base` plus `focus` are optional.
- D-03: User-facing instructions must ask for prioritized concrete findings, not long narrative critique.
- D-04: Team rollout docs must present slash commands first and MCP tools as reference underneath.
- D-05: Docs and prompts must not imply automatic full Codex chat transfer.
- D-06: Docs and prompts must mention `git status` and untracked files.
- D-07: Docs must name selected current-phase planning docs rather than every `.planning/**` file.
- D-08: Docs must define `resume` as Claude Code session continuity only.
- D-10: `claude_rescue` and `allow_write` must be documented only as outside the default review path.
- D-11: Docs/prompts must preserve the read-only/no-files-edited result expectation.
- D-12: Hooks must stay outside default onboarding and carry loop, usage-cost, and blocking-risk warnings.
- D-14: Docs/prompts must use `High`, `Medium`, and `Low`.
- D-15: Docs/prompts must explain `No high-confidence findings`.
- D-16: Docs must list actionable failure categories.
</must_haves>
