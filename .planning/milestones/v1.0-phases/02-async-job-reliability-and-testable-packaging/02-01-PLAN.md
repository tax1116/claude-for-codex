# Plan 02-01: Fake-Claude Job Lifecycle Contracts

<objective>
As a maintainer, I want deterministic tests around background Claude jobs so
that status, result, cancel, parsing, timeout, and launch-failure behavior can
be trusted without a live Claude account.
</objective>

<must_haves>
- D-01: Long-running review tools can start in background and return a task id immediately.
- D-02: Status lists running and recent jobs for the current repository.
- D-03: Result fetches final output and includes Claude session id guidance when present.
- D-04: Cancel marks a running job cancelled while the current MCP server owns the child process.
- D-05: Cancel output states the process-lifetime limit and avoids durable queue promises.
- D-06: Fake-Claude tests cover Claude-style JSON parsing and session id persistence.
- D-07: Fake-Claude tests cover text fallback for malformed JSON output.
- D-08: Fake-Claude tests cover non-zero exits and launch failures.
- D-09: Timeout behavior is testable with a short test timeout.
- D-10: New job behavior goes through StateStore; no new ad hoc job-file helpers.
</must_haves>

## Artifacts this plan produces

- Testable job runner seam or helper module, only if needed.
- `test/job-lifecycle.test.mjs` fake-Claude integration coverage.
- Updated status/result/cancel output where product wording is missing.
- StateStore usage kept as the only persistence boundary.
