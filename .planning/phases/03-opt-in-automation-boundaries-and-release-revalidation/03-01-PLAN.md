# Plan 03-01: Hook And Rescue Safety Boundaries

<objective>
As a teammate adopting the plugin, I want hook automation and write-capable
rescue to be unmistakably advanced, optional, reversible, and risky so that I
do not enable automation or write permissions by accident.
</objective>

<must_haves>
- D-17: Default onboarding remains manual slash-command review; hook setup is not
  part of the default install path.
- D-18: Hook docs state opt-in setup and clear rollback/disable steps.
- D-19: Hook docs state loop risk, completion-blocking risk, and usage-cost risk.
- D-20: `claude_rescue allow_write` remains outside the standard v1 review path.
- D-21: Any write-capable rescue docs warn about broad write permissions and
  `--dangerously-skip-permissions`.
</must_haves>

## Artifacts this plan produces

- Stronger docs contract tests for hook opt-in/reversibility/risk language.
- Runtime/source contract tests for read-only review versus write-capable rescue.
- Updated README/setup/design wording around hooks and rescue.
- Optional hook source comment wording update if docs/tests need consistency.
