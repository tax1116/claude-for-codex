# Plan 03-02: Release Revalidation Markers

<objective>
As a release reviewer, I want time-sensitive setup, hook, model, billing, and
package claims marked for release-date revalidation so that v1 docs do not ship
stale guidance as if it were permanent truth.
</objective>

<must_haves>
- D-22: Release-facing docs identify time-sensitive claims that must be checked
  near release.
- D-23: Revalidation markers cover Codex CLI/MCP config, hook behavior, Claude
  Code CLI behavior, model aliases, billing/Agent SDK usage, and npm/package
  setup.
- D-24: Publishing docs align with the repo's current `dev` to `master` PR
  promotion policy.
- D-25: Docs avoid claiming current external behavior unless execution has
  checked official docs for the release date.
</must_haves>

## Artifacts this plan produces

- Release revalidation checklist/table in release-facing docs.
- Docs tests that lock the checklist categories.
- Publishing docs aligned with `docs/BRANCHING.md`.
- Notes on which claims need official OpenAI/Anthropic/npm/GitHub verification
  during release work.
